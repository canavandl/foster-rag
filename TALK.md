# Foster RAG — 3-Minute Technical Walk-Through

> Draft script for job application prompt:
> *"Walk through a technical project you have worked on. What was the use case? Was there an interesting challenge? How did you solve it? What was the tech stack?"*

---

## [0:00 — Hook & Use Case]

My wife and I became foster parents last year. It's been a great experience, but one of the challenges is that there are a huge number of state regulations that we need to follow related to things like our home and visitors, and the child's health and education. Texas foster care regulations are spread across dozens of PDFs — primarily the CPS Policy Handbook and various official Resource Guides — and getting the correct answer is important because I'd hate to be out of compliance due to a hallunicated LLM response. Fortunately - this is the kind of problem that retrieval augmented systems are good at, where I can ask natural language questions about regulations and get accurate answers backed by citations to the actual regulatory text. 

---

## [0:30 — Tech Stack]

"The entire stack runs on Cloudflare's edge platform. The compute layer is a **Cloudflare Worker** written in TypeScript. Documents and chunks are stored in **Cloudflare D1** — their serverless SQLite offering. Vector search is powered by **Cloudflare Vectorize**, with 768-dimensional embeddings generated by the `bge-base-en-v1.5` model. Answer generation uses `llama-3.1-8b-instruct` — both models served via **Cloudflare Workers AI**.

The reason I picked this stack: zero infrastructure management, everything co-located at the edge, and a unified billing surface. The entire thing deploys with `wrangler deploy`."

---

## [1:00 — How It Works]

"The pipeline has two phases.

**Ingestion**: I extract text from PDFs using `pdf-parse`, inject page markers so I can track which page each chunk came from, then split into 512-token chunks with 50-token overlap. Each chunk gets embedded and stored three ways: the vector goes into Vectorize, the text and metadata go into D1.

**Query**: When a user asks a question, I embed the query, run a vector similarity search scoped by namespace — say, `foster_parent` — retrieve the top 5 matching chunks from D1, pass them as context to the LLM, and return the answer along with source citations that include the document name, section, similarity score, and a direct PDF page link."

---

## [1:45 — Interesting Challenge]

"The most interesting design decision was **namespace filtering without multiple indexes**.

My initial instinct was to create separate Vectorize indexes per audience — one for caseworkers, one for foster parents, one for legal. But Vectorize charges per index and adds operational overhead. Instead, I store a `namespace` field in each vector's metadata and use Cloudflare's metadata filtering at query time. One index, logical separation, no extra cost.

The other challenge was **page-level citation accuracy**. PDF extraction loses page boundaries by default. I solved this by injecting `<<<PAGE:N>>>` markers during extraction and tracking the starting page number for each chunk. So every answer links directly to the exact page in the source PDF — like `chapter-749.pdf#page=47`."

---

## [2:30 — Results & Wrap-Up]

"I built an evaluation suite with 13 test cases covering all five namespaces and cross-cutting queries. Each test checks three things: did the expected source documents appear, did the answer contain the expected keywords, and did any wrong-namespace documents bleed in.

Current results: 12 out of 13 passing, 93.6% average source recall, 100% namespace isolation.

The project is live on Cloudflare Workers and the code is open source at github.com/canavandl/foster-rag. I'd call Phase 1 done — the obvious next step is semantic chunking to push source recall toward 100%."
